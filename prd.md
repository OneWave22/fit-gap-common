# PRD: 핏갭 (Fit-Gap) — AI 채용 분석 플랫폼

버전: 1.0
작성일: 2026-02-06
상태: Draft

## 1. 개요

### 1.1 제품 한 줄 요약

> 공고(JD)와 서류(자소서/포폴) 사이의 Fit-Gap을 분석하여, 구직자에게는 개선 방향을, 기업에게는 스크리닝 근거를 제공하는 양방향 채용 분석 플랫폼

### 1.2 배경

- 구직자는 평균 수십 곳에 지원하지만 탈락 사유를 알 수 없어 방향성을 상실한다.
- 기업은 대량의 지원서를 수동 검토하느라 채용 피로도가 높고, 공고 자체의 문제를 인지하지 못한다.
- 기존 서비스(사람인/원티드/ChatGPT 첨삭)는 "추천"은 하되 "왜"를 설명하지 않거나, "첨삭"은 하되 "공고 맥락"을 모른다.

### 1.3 목표

| 구분 | 목표 |
|------|------|
| **제품 목표** | 공고-서류 간 Fit/Gap/Over를 구조적으로 분석하고, 근거와 함께 양방향 피드백 제공 |
| **해커톤 MVP 목표** | 백엔드 개발 직무 1개에 대해 Fit-Gap 분석 + 신호등 분류 + 공고 개선 인사이트 데모 |
| **비즈니스 목표** | B2C Free로 구직자 유입 → 데이터 축적 → B2B 스크리닝 과금(양면 네트워크 효과) |

### 1.4 성공 지표 (KPI)

| 지표 | MVP 목표 | 출시 6개월 목표 |
|------|----------|----------------|
| 분석 완료율 (서류 업로드 → 결과 확인) | 데모 성공 | 80% 이상 |
| 피드백 유용성 (👍 비율) | - | 70% 이상 |
| 재방문율 (7일 내 재사용) | - | 40% 이상 |
| B2B 전환율 (무료 체험 → 유료) | - | 10% 이상 |

## 2. 사용자 정의

### 2.1 구직자 (B2C)

- **프로필**: 26~28세, 신입 취준 6~12개월, IT 직군(백엔드/프론트 개발)
- **핵심 Pain**: 다수 지원 후 탈락 사유를 모름. GPT 첨삭은 공고 맥락이 없음.
- **핵심 Need**: 특정 공고 기준으로 "왜 안 맞는지" + "어떻게 고쳐야 하는지" 피드백

### 2.2 기업 HR (B2B)

- **프로필**: 32~36세, 20~60명 스타트업 HR 매니저
- **핵심 Pain**: 지원서 200개 수동 검토, 공고 품질 검증 불가, 피드백 제공 여력 없음
- **핵심 Need**: 상위 후보 자동 스크리닝 + 공고 개선 인사이트

## 3. 기능 요구사항

### 3.1 MVP (해커톤) 범위

#### F-01. 서류 업로드 및 파싱

| 항목 | 내용 |
|------|------|
| **설명** | 구직자가 자소서 PDF를 업로드하면 텍스트를 추출하고 구조화된 데이터로 변환 |
| **입력** | PDF 파일 (텍스트 기반) |
| **출력** | 구조화된 서류 데이터: {보유기술[], 프로젝트경험[], 성과지표[], 소프트스킬[], 키워드[]} |
| **우선순위** | P0 (필수) |

#### F-02. 공고 입력 및 파싱

| 항목 | 내용 |
|------|------|
| **설명** | 구직자 또는 기업이 공고 텍스트를 입력하면 구조화된 데이터로 변환 |
| **입력** | 공고 텍스트 (붙여넣기) |
| **출력** | 구조화된 공고 데이터: {필수기술[], 우대기술[], 핵심업무[], 요구경험[], 조직문화키워드[]} |
| **우선순위** | P0 (필수) |

#### F-03. Fit-Gap 분석 엔진

| 항목 | 내용 |
|------|------|
| **설명** | 공고와 서류의 구조화 데이터를 교차 분석하여 Fit/Gap/Over를 분류하고 적합도 점수를 산출 |
| **분석 파이프라인** | Step 1. 구조화 추출 → Step 2. 의미 단위 매칭(임베딩 유사도) → Step 3. Gap 분류 → Step 4. 점수 산출 + 설명 생성 |
| **점수 산출** | 필수기술(35%) + 우대사항(15%) + 직무경험(25%) + 소프트스킬(15%) + 성과근거(10%) |
| **출력** | 적합도 점수(0~100) + 항목별 점수 + 부족 TOP 3 + 강점 TOP 3 + 원문 인용 설명 |
| **우선순위** | P0 (필수) |

#### F-04. 신호등 분류

| 항목 | 내용 |
|------|------|
| **설명** | 적합도 점수 기반 3단계 자동 분류 |
| **기준** | 0~40%: 🔴 부적합, 40~80%: 🟡 보류, 80~100%: 🟢 적합 |
| **출력** | 분류 라벨 + 부족 사유 1줄 요약 |
| **우선순위** | P0 (필수) |

#### F-05. 공고 개선 인사이트 (기업용)

| 항목 | 내용 |
|------|------|
| **설명** | 공고 텍스트를 분석하여 개선 제안 2~3개 제공 |
| **분석 항목** | 필수 요건 과다 여부, 신입/경력 불일치 탐지, 복지/문화 기술 부족 |
| **출력** | 인사이트 카드 2~3개 (문제점 + 개선 제안 + 근거) |
| **우선순위** | P0 (필수) |

#### F-06. 서류 개선 제안 (구직자용)

| 항목 | 내용 |
|------|------|
| **설명** | Fit-Gap 분석 결과에서 도출된 Gap 항목별 구체적 개선 방향 제시 |
| **출력** | Gap 항목별 "무엇을 추가/수정해야 하는지" 제안 + 원문 인용 근거 |
| **우선순위** | P1 (높음) |

### 3.2 확장 기능 (MVP 이후)

## 5. 기술 아키텍처

### 5.1 시스템 구성

```
[클라이언트]                    [서버]                         [외부 서비스]
React + Tailwind  ──HTTP──▶  FastAPI                        OpenAI GPT-4o
                              ├─ /api/resumes    (서류 파싱)     (구조화 추출, 설명 생성)
                              ├─ /api/postings   (공고 파싱)   OpenAI Embeddings
                              ├─ /api/analysis   (Fit-Gap)     (의미 유사도)
                              ├─ /api/screening  (신호등)     Supabase (PostgreSQL)
                              └─ /api/insights   (공고 인사이트)  (데이터 저장)
```

### 5.2 기술 스택

| 레이어 | 기술 | 선택 이유 |
|--------|------|-----------|
| 프론트엔드 | React + Tailwind CSS | 빠른 프로토타이핑 |
| 백엔드 | FastAPI (Python) | LLM 연동 생태계, 빠른 개발 |
| AI/LLM | OpenAI GPT-4o (또는 Claude) | Structured Output 지원 |
| 임베딩 | OpenAI text-embedding-3-small | 의미 유사도 비교 |
| DB | Supabase (PostgreSQL) | 쿼리 부담 감소, 배포 불필요 |
| 배포 | Vercel (프론트) + Fly.io (백엔드) | 무료 티어 활용 |
| PDF 파싱 | pdfplumber | 텍스트 추출 |

### 5.3 Fit-Gap 분석 파이프라인

```
Step 1. 구조화 추출 (LLM Structured Output)
  - 입력: 원문 텍스트
  - 출력: JSON (사전 정의 스키마)
  - LLM 호출 1회 (공고), 1회 (서류)

Step 2. 의미 단위 매칭 (Embedding Similarity)
  - 공고 요구사항 각 항목 ↔ 서류 역량 각 항목 임베딩 비교
  - 유사도 임계값: 0.75 이상 → Fit, 0.5~0.75 → 부분매칭, 0.5 미만 → Gap

Step 3. Gap 분류 (규칙 기반)
  - Fit: 유사도 0.75+ 매칭 존재
  - Gap: 공고 요구 항목에 매칭 서류 항목 없음
  - Over: 서류 항목에 매칭 공고 항목 없음 + 톤 불일치

Step 4. 점수 산출 (규칙 기반) + 설명 생성 (LLM)
  - 가중 합산으로 적합도 점수 계산
  - LLM 호출 1회: Gap/Fit 분류 결과 기반 설명문 생성 (원문 인용 포함)
```

## 6. 데이터 모델

### 6.1 핵심 엔티티

```
Resume (서류)
├─ id: UUID
├─ user_id: UUID
├─ raw_text: TEXT           # PDF 추출 원문
├─ parsed_data: JSONB       # 구조화 추출 결과
├─ created_at: TIMESTAMP
└─ is_stored: BOOLEAN       # 사용자가 저장을 선택했는지

JobPosting (공고)
├─ id: UUID
├─ company_name: VARCHAR
├─ raw_text: TEXT           # 공고 원문
├─ parsed_data: JSONB       # 구조화 추출 결과
├─ created_by: UUID         # 기업 사용자 또는 구직자
└─ created_at: TIMESTAMP

Analysis (분석 결과)
├─ id: UUID
├─ resume_id: UUID (FK)
├─ posting_id: UUID (FK)
├─ overall_score: INTEGER   # 0~100
├─ category_scores: JSONB   # 항목별 점수
├─ fit_items: JSONB         # 일치 항목 리스트
├─ gap_items: JSONB         # 부족 항목 리스트
├─ over_items: JSONB        # 과잉 항목 리스트
├─ explanation: TEXT         # 종합 설명문
├─ confidence: VARCHAR      # High/Medium/Low
├─ created_at: TIMESTAMP
└─ feedback: VARCHAR        # 사용자 피드백 (thumbs_up/thumbs_down/null)

PostingInsight (공고 인사이트)
├─ id: UUID
├─ posting_id: UUID (FK)
├─ insights: JSONB          # 인사이트 카드 배열
└─ created_at: TIMESTAMP
```

## 7. 사용자 플로우

### 7.1 구직자 플로우 (MVP)

```
서류 PDF 업로드
    → AI 파싱 결과 확인/수정
        → 관심 공고 텍스트 입력 (1~5개)
            → Fit-Gap 분석 실행
                → 결과 확인: 적합도 점수 + 신호등 + Fit/Gap/Over 상세
                    → 개선 제안 확인
                        → (선택) 서류 수정 후 재분석
```

### 7.2 기업 플로우 (MVP)

```
공고 텍스트 입력
    → 공고 개선 인사이트 확인
        → (선택) 공고 수정
            → 지원자 서류 업로드 (다건)
                → 자동 Fit-Gap 분석 + 신호등 분류
                    → 지원자 리스트 (🟢🟡🔴) + 부족 사유 확인
                        → (선택) 탈락자 피드백 메일 생성
```

## 9. 제약 조건 및 가정

### 제약 조건

- 해커톤 MVP는 **백엔드 개발 직무 1개**만 지원
- PDF는 텍스트 기반만 지원 (이미지/스캔 PDF는 미지원)
- 공고 데이터는 사용자 직접 입력 (크롤링/API 연동 없음)
- 합격 통계("상위군 평균")는 MVP에서 제공하지 않음

### 가정

- 구직자는 자소서를 PDF 형태로 보유하고 있다
- 공고 텍스트는 공개 정보로, 구직자가 복사/붙여넣기 가능하다
- LLM의 구조화 추출 정확도가 실용 수준(90%+)이다
- 사용자는 분석 결과의 "참고용" 성격을 이해한다
