# 서비스명 : 핏갭 (Fit-Gap)

: "합격과 불합격 사이, 그 1%의 이유를 찾아주는 AI 채용 분석관"

# 1) 서비스 핵심 컨셉 요약 (IR 한 문장)

> **합격과 불합격 사이, 그 간극을 설명하는 양방향 채용 분석 플랫폼**

"매칭 플랫폼"이 아니라, 공고(JD)와 서류(자소서/포폴) 사이의 '핏-갭(Fit-Gap)'을 분석해
구직자에게는 *어디를 넣을지 + 왜 떨어지는지 + 무엇을 고쳐야 하는지*, 기업에게는 *누굴 봐야 하는지 + 공고를 어떻게 고쳐야 하는지*를 주는 '설명 가능한 채용 전략 서비스'

---

# 2) 문제정의

### Problem : 정보의 비대칭과 블랙박스

- **구직자의 Pain:** "100군데 지원해서 다 떨어졌는데, 스펙이 문제인지 자소서가 문제인지 아무도 안 알려준다." (피드백의 부재 → **방향성 상실**)
- **기업의 Pain:** "지원자는 200명인데, 우리 회사 핏(Fit)에 맞는 사람은 찾기 힘들고 검토할 시간도 없다." (검증 비용 과다 → **채용 피로도**)
- **정의:** 취업난은 단순한 일자리 부족이 아니라, **서로의 눈높이를 확인하고 조율할 '피드백 데이터'가 없는 깜깜이 시장**이기 때문에 발생합니다.

### 기존 서비스 대비 차별점 (경쟁 분석)

| 기존 서비스 | 하는 일 | 한계 | Fit-Gap의 차별점 |
|---|---|---|---|
| **사람인/잡코리아 AI 매칭** | 이력서 키워드 기반 공고 추천 | 단순 키워드 매칭, "왜 이 공고인지" 설명 없음 | 공고-서류 간 의미 단위 교차 분석 + 부족/과잉 사유 설명 |
| **원티드** | 이력서 기반 기업 추천 + 합격보상금 | 추천 근거 블랙박스, 서류 개선 피드백 없음 | 적합도 점수의 산출 근거를 항목별로 투명 공개 |
| **ChatGPT 자소서 첨삭** | 범용 문장 교정 | 특정 공고(JD) 맥락 없이 일반적 조언만 제공 | 지원 대상 공고 기준의 맞춤 분석(JD-Aware) |
| **Lever/Greenhouse (ATS)** | 지원자 파이프라인 관리 | 스크리닝은 수동, 분석/피드백 기능 없음 | AI 자동 스크리닝 + 공고 개선 인사이트 |

**핵심 차별점 요약:** 기존 서비스들은 "추천"은 하되 "왜"를 설명하지 않고, "첨삭"은 하되 "공고 맥락"을 모른다. Fit-Gap은 공고(JD)와 서류를 교차 분석하여 **근거 있는 설명**을 양방향으로 제공하는 유일한 서비스다.

# 3) 핵심 기능 정리 (해커톤 MVP 기준)

## A. 공통 핵심 엔진 (서비스의 '차별점'이 되는 부분)

### 1. 공고-서류 **교차 분석(Fit-Gap 분석)**

- 공고(필수/우대/업무)에서 핵심 요구를 추출
- 서류(자소서/포폴)에서 경험·역량·키워드를 추출
- 둘의 **일치/부족/과잉**을 분류해서 기업과 구직자에게 분리해서 보여줌

#### 분석 파이프라인 (기술 상세)

```
[입력: 공고 텍스트 + 서류 텍스트]
    ↓
Step 1. 구조화 추출 (LLM Structured Output)
  - 공고 → {필수기술[], 우대기술[], 핵심업무[], 요구경험[], 조직문화키워드[]}
  - 서류 → {보유기술[], 프로젝트경험[], 성과지표[], 소프트스킬[], 키워드[]}
    ↓
Step 2. 의미 단위 매칭 (Semantic Matching)
  - 단순 문자열 매칭이 아닌, 임베딩 기반 유사도 비교
  - 예: "Spring Boot 실무 2년" ↔ "Spring Framework 기반 API 개발 경험"은 매칭
  - 예: "팀 리딩 경험" ↔ "사이드 프로젝트 팀장"은 부분 매칭
    ↓
Step 3. Gap 분류
  - ✅ Fit (일치): 공고 요구 ↔ 서류 근거가 명확히 대응
  - ⚠️ Gap (부족): 공고가 요구하나 서류에 근거 없음
  - 📌 Over (과잉): 서류에 있으나 공고와 무관하거나 톤 불일치
    ↓
Step 4. 점수 산출 + 설명 생성
  - 항목별 가중 합산 → 적합도 점수
  - 각 Gap/Fit 항목에 대해 "왜 이렇게 판단했는지" 원문 인용 기반 설명 생성
```

#### 적합도 점수 산출 체계

| 평가 항목 | 가중치 (기본값) | 설명 |
|---|---|---|
| 필수 기술 스택 일치도 | 35% | 공고 필수 요건 vs 서류 기술 경험 |
| 우대 사항 충족도 | 15% | 공고 우대 요건 vs 서류 추가 역량 |
| 직무 경험 관련도 | 25% | 핵심 업무 vs 프로젝트/경력 경험 |
| 소프트스킬·문화 적합도 | 15% | 조직문화 키워드 vs 서류 톤·표현 |
| 성과·수치 근거 충실도 | 10% | 정량적 성과 제시 여부 |
| **합계** | **100%** | |

> 참고: 직무별 가중치 프리셋을 제공한다. 예) 백엔드 개발: 필수기술 40%, 직무경험 30% / PM: 소프트스킬 25%, 직무경험 35%. 기업은 가중치를 직접 조정할 수도 있다.

**출력 형태 (데모에 강함)**

- 적합도 점수(0~100) + 산출 근거 항목별 점수
- 부족 요소 TOP 3 (예: Redis 실무, 협업 사례, 장애 대응)
- 강점 요소 TOP 3
- "왜 이렇게 나왔는지" 한 문단 설명 — **반드시 원문 인용(citation) 포함**

### 2. 지원자 및 기업 자동 스크리닝(신호등/랭킹)

- 🔴 부적합 / 🟡 보류 / 🟢 적합
    - 0%~40%: 🔴 부적합
    - 40%~80%: 🟡 보류
    - 80%~100%: 🟢 적합
- 점수만이 아니라 "부족 사유"까지 카드로 제공

## B. 구직자 기능 (B2C)

### 1. 내 서류 약점 진단 + **개선 방향(Actionable Insight)**

- "문장 다듬기"가 아니라 "공고 기준으로" 고치게 함
- 예: "협업 경험 언급 1회 → 합격 상위군 평균 3회"

### 2. **역방향 매칭(Reverse Matching)**: 내 서류에 맞는 공고/기업 추천

- "추천 이유 + 비추천 이유" 동시 제공
- "지원 우선순위"를 만들어 줌 (전략 키워드에 딱 맞음)

> **MVP 범위 주의:** 역방향 매칭은 충분한 공고 DB가 전제된다. 해커톤 MVP에서는 사용자가 직접 관심 공고 3~5개를 입력하면 그 중에서 우선순위를 매겨주는 방식으로 축소 구현한다. 대규모 공고 크롤링/DB 구축은 고도화 단계에서 진행.

### 3. **핏-갭(Fit-Gap) 분석 리포트**

> [분석 결과]
> ✅ 강점: 기술 스택 90% 충족 (Spring Boot, AWS)
> ⚠️ 약점: 협업 경험 언급 1회 (상위 10% 합격자 평균 3회)
> 📊 부족 키워드: Redis, 트러블슈팅 사례
> 📈 과한 키워드: 모험, 도전 (이 기업은 '안정' 중시)

- 단순 점수가 아닌, 공고(JD) 대비 내 서류의 **부족한 키워드(Gap)**와 **과한 키워드(Over)**를 시각화합니다.
- *예: "이 공고는 '협업'을 5번 언급했는데, 지원자님은 '개인 성과'만 강조하고 있습니다."*

### 4. 기업 자동 스크리닝(신호등/랭킹)

- 🔴 비추천 / 🟡 보류 / 🟢 추천
    - 0%~40%: 🔴 비추천
    - 40%~80%: 🟡 보류
    - 80%~100%: 🟢 추천

### (선택) 5. **합격 가능성 시뮬레이션**

- "이 키워드를 보완하면 점수가 얼마나 오르는지" 가시화
    - "이 키워드를 추가하면 A사 적합도 72% → 89%로 상승"
- 실시간 수정 시뮬레이션 제공
    - 해커톤 데모에 가장 임팩트 있음 (Before/After)

### (선택) 6. AI 면접관의 예상 질문 (Pre-Interview Warning, 고도화)

- 서류 제출 전, AI가 예상 면접 질문 + 약점 공격
- 예시: "이 자소서대로라면 면접관은 **'이 프로젝트에서 본인의 구체적인 기여도가 무엇인가요?'**라고 100% 물어볼 것입니다"
- **효과**: 서류 제출 전 스스로 검증하고 보완

## C. 기업 기능 (B2B)

### 1. 공고 개선 인사이트 (기업도 검증받는 구조)

- "신입 유입이 낮을 공고" 경고
- "필수/우대 요구 과도" 알림
- "복지 항목 강화 시 지원자 풀이 넓어짐" 같은 제안

### 2. 지원자 자동 스크리닝(신호등/랭킹)

- 🔴 부적합 / 🟡 보류 / 🟢 적합
    - 0%~40%: 🔴 부적합
    - 40%~80%: 🟡 보류
    - 80%~100%: 🟢 적합

### (선택) 3. 원클릭 피드백 레터 생성(탈락 메일 자동 작성)

- 기업의 시간 절감 + 채용 브랜딩(좋은 거절)
- 탈락 버튼을 누르면, AI가 해당 지원자의 서류 내용을 바탕으로 **"구체적인 불합격 사유가 담긴, 세상에서 가장 정중한 거절 메일"**을 자동 생성합니다.
- *기대 효과:* 기업의 채용 브랜딩(이미지) 상승 + 구직자에게 실질적 도움.
- 해커톤에서 "와 이거 쓸만한데?"를 만들기 쉬움

# 4) AI 피드백 신뢰도 확보 전략 (신규 섹션)

> AI 기반 채용 분석의 최대 리스크는 "그 피드백을 왜 믿어야 하는가?"이다. 아래 5가지 전략으로 신뢰도를 확보한다.

## 전략 1. 원문 인용 기반 설명 (Citation-Based Explanation)

**원칙: 모든 분석 결과에는 반드시 원문 근거를 첨부한다.**

- 피드백의 모든 항목은 "공고의 어느 문장"과 "서류의 어느 문장"을 비교한 결과인지 명시
- 사용자가 직접 원문을 확인하고 AI 판단의 타당성을 검증할 수 있게 함

```
[나쁜 예] "협업 경험이 부족합니다." → 근거 없음, 신뢰 불가

[좋은 예]
  Gap 항목: 협업 경험
  공고 원문: "크로스펑셔널 팀과의 협업 경험 필수" (필수 요건 3번)
  서류 분석: 4개 프로젝트 서술 중 팀 협업 언급 1회,
            구체적 협업 방식(코드리뷰, 스프린트 등) 서술 0회
  판단: 공고가 요구하는 협업 수준 대비 근거 불충분 → Gap
```

## 전략 2. 구조화된 평가 루브릭 (Structured Rubric)

**원칙: LLM의 자유 생성이 아닌, 사전 정의된 평가 프레임워크 위에서 분석한다.**

- 직무별 평가 기준 템플릿 사전 정의 (백엔드/프론트/PM/디자인 등)
- 각 기준별 가중치와 판단 기준을 투명하게 공개
- LLM은 "판단"이 아니라 "추출 + 분류"를 수행하고, 점수 산출은 규칙 기반으로 처리

```
[LLM 역할 분리]
- LLM이 하는 것: 텍스트에서 정보 추출, 의미 유사도 비교, 설명문 생성
- LLM이 하지 않는 것: 최종 점수 산출 (규칙 기반 엔진이 담당)
- 효과: LLM hallucination이 점수에 직접 반영되는 것을 방지
```

## 전략 3. 확신도 표시 + 불확실성 고지 (Confidence Indicator)

**원칙: AI가 확신하지 못하는 부분은 솔직하게 표시한다.**

- 각 분석 항목에 확신도(High/Medium/Low)를 함께 표시
- 확신도가 Low인 항목은 "참고용"으로 표시하고, 사용자에게 직접 판단을 권유
- 적합도 점수는 단일 값이 아닌 **범위**로 표시 (예: 72점 ± 8점)

```
[예시 출력]
  ✅ 기술 스택 일치도: 88점 (확신도: High)
     → 공고 필수 5개 중 4개 명확히 매칭
  ⚠️ 소프트스킬 적합도: 55점 (확신도: Low)
     → 공고의 "리더십" 요구와 서류의 "팀 프로젝트 경험"이
        어느 수준까지 매칭되는지 텍스트만으로 판단 어려움
     → [참고용] 면접에서 구체적 리더십 사례를 준비하는 것을 권장
```

## 전략 4. 재현성 검증 (Consistency Check)

**원칙: 같은 입력에 같은 결과가 나와야 한다.**

- 동일한 공고+서류 조합을 복수 회 분석해 결과 편차 측정
- Temperature를 낮추고, 구조화된 출력(JSON Schema)을 강제하여 응답 일관성 확보
- 편차가 큰 항목은 자동으로 확신도를 낮춤

## 전략 5. 피드백 루프 (사용자 검증)

**원칙: 사용자의 이의 제기를 반영하여 분석을 개선한다.**

- 각 분석 항목에 "동의/비동의" 버튼 제공
- 비동의 시 사유 입력 → 해당 케이스를 수집하여 프롬프트/루브릭 개선에 활용
- 장기적으로 직무별 "오분석 패턴" DB를 구축하여 정확도 향상

```
[MVP에서의 적용]
- 분석 결과 하단에 "이 분석이 도움이 되었나요?" (👍/👎)
- 👎 선택 시 → "어떤 부분이 부정확했나요?" 선택지 제공
  (예: "해당 기술을 보유하고 있음", "경험을 다르게 해석함" 등)
```

# 5) 데이터 소싱 전략 (신규 섹션)

## MVP 단계 (해커톤)

| 데이터 | 소싱 방법 | 비고 |
|---|---|---|
| 공고(JD) | 사용자 직접 입력 (텍스트 붙여넣기 또는 URL) | 크롤링 불필요, 즉시 구현 가능 |
| 서류 (자소서/포폴) | 사용자 PDF 업로드 → LLM 파싱 | OCR 불필요 (텍스트 기반 PDF) |
| 합격 통계 ("상위군 평균") | **MVP에서는 제공하지 않음** | 검증 불가능한 통계는 신뢰도를 해침 |
| 직무별 평가 기준 | 팀 내 수동 작성 (백엔드 1개 직무) | 해커톤은 1개 직무만 깊이 있게 |

## 고도화 단계

| 데이터 | 소싱 방법 |
|---|---|
| 공고 DB | 채용 플랫폼 공개 API / RSS 활용 (사람인 오픈API 등) |
| 합격 통계 | 서비스 내 누적 데이터(익명화) + 사용자 합격/불합격 결과 자발적 제보 |
| 직무별 기준 | HR 전문가 자문 + 서비스 데이터 기반 자동 학습 |

> **중요:** MVP에서 "합격 상위군 평균 3회" 같은 통계를 언급하려면, 반드시 "참고 데이터 출처: ○○ 조사 기반"을 명시하거나, 해당 통계를 제거해야 한다. 근거 없는 숫자는 서비스 전체의 신뢰를 무너뜨린다.

# 6) 고도화(Advanced) 로드맵 제안

해커톤 후 IR에서 "우리는 여기까지 간다"를 보여주는 용도.

## 1단계: 정교한 Fit-Gap (정확도 업)

- 직무별 평가 기준 템플릿(백엔드/프론트/PM 등)
- "키워드 매칭" → "맥락 매칭(경험-요구 연결성)" 강화

## 2단계: 데이터 축적 기반 인사이트

- 직무별 합격 상위군 패턴(익명화)
- "이 직무 합격자들이 공통으로 쓰는 구조/근거 방식" 제공

## 3단계: 기업용 ATS 플러그인(B2B 확장)

- 지원자 정렬/클러스터링
- 인터뷰 질문 자동 생성(핏-갭 기반)
- 채용 회의용 요약 리포트 자동 생성

## 4단계: 조직/문화 핏 확장(중견 이상)

- 팀 문화/업무 방식 입력 → 문화 핏 분석
- 조기 퇴사 리스크 감소까지 확장

# 9) 타겟 고객별 페르소나 정리 (구직자 1개, 기업 1개)

---

## 페르소나 1 (Primary, 구직자): 김준비

- **나이/상태**: 26~28세 / 신입 취준 6~12개월
- **직무**: 백엔드(또는 프론트) 개발
- **스펙:** 학점 3.8, 토익 850, 부트캠프 수료
- **상황**: 지원 73곳 → 서류 통과 8곳 → 최종 합격 0곳
- **Pain**
    - "프로젝트도 했고, 코딩테스트도 통과했는데 왜 떨어질까?"
    - "GPT 첨삭을 받아도 이게 공고에 맞는지 모르겠다"
    - "자소서를 50번 고쳤는데도 똑같이 떨어져요"
    - "친구는 스펙이 나보다 낮은데 붙었어요. 뭐가 다른 거죠?"
- **Needs**
    - 공고 기준으로 "왜 안 맞는지"를 알고 싶다 → 개선 가능한 피드백
    - 구체적인 탈락 이유
    - 어디에 지원해야 효율적인지(전략)
- **우리 서비스가 주는 변화**
    - 무지성 지원 → **갭 기반 수정 + 우선순위 지원**
- **서비스 사용 시나리오**
    ```
    1. 자기소개서 PDF를 업로드 한다.
    2. 서비스 AI가 PDF를 읽어서 분석 및 추출해 각 내용 필드에 알아서 입력
    3. 관심 공고 URL 또는 텍스트를 입력한다. (MVP: 직접 입력)
    4. Fit-Gap 분석 결과 확인:
       - 적합도 점수 + 항목별 상세 점수
       - 부족 요소 TOP 3 (원문 인용 포함)
       - 강점 요소 TOP 3
    5. A사 적합도 92%(🟢), B사 68%(🟡) 확인
    6. "B사에 지원하려면 'Redis 경험' 추가 필요" 개선안
    7. 키워드 및 경험 개선해 자기소개 수정 → 적합도 68% → 85% 상승
    8. 사이트 내에서 자기소개서 전략적 수정 후 다음 공고 지원
    ```
- **가장 설득력 있는 데모**
    - (Before) A사 적합도 62, 부족 TOP3 + 원문 근거
    - (After) 수정 후 84로 상승 + 수정 포인트가 명확

## 페르소나 2 (Primary, 기업): 스타트업 HR/팀리드 박답답

- **나이/상태**: 32~36세 / 20~60명 시리즈A 스타트업 HR 매니저
- **채용 포지션**: 60명 규모, 백엔드 3명 채용 목표 (즉시 투입 희망)
- **Pain**
    - 지원서 200개 중 "제대로 된" 지원자 찾기 어려움
    - 서류 검토에 시간 소모(일주일), 면접 봤는데 실력이 기대 이하면 시간 낭비
    - 피드백을 주고 싶어도 여력 없음 (+브랜딩 우려)
- **Needs**
    - 상위 후보만 빠르게 추려주는 스크리닝
    - 공고 자체가 문제인지 확인하는 인사이트
- **우리 서비스가 주는 변화**
    - 수동 검토 → **자동 분류 + 요약 카드 + 공고 개선**
- **서비스 사용 시나리오**
    ```
    1. 공고 초안 등록
      (예시 : 지원자 조건 - Spring Boot, Redis, AWS, Spring Batch 필수)
    2. AI가 공고 텍스트를 분석 → 공고 개선 인사이트 제공
       (예: "필수 요건 4개는 경력 3년차 수준. 신입 대상이라면 2개로 축소 권장")
    3. 피드백 바탕으로 필요한 부분 수정 후 게시
    4. 지원자 서류를 업로드하면 자동 Fit-Gap 분석
    5. 신호등 기능: 🟢적합(3명), 🟡보류(12명), 🔴부적합(35명)
    6. 상위 10명만 면접 → 시간 80% 절감
    7. 탈락자에게 원클릭 피드백 메일 자동 발송 (선택)
    ```
- **가장 설득력 있는 데모**
    - 지원자 리스트에 🔴🟡🟢 분류 + "부족 이유 1줄" + 근거 원문
    - 공고 개선 제안 2~3개(신입 유입 개선)
