### UC-01: 서류 업로드 및 파싱

**액터**: 구직자
**우선순위**: P0 (MVP 필수)

**사전 조건**
- 구직자가 텍스트 기반 PDF 형태의 자소서를 보유하고 있다.

**기본 플로우**

| 단계 | 액터 | 행동 |
|------|------|------|
| 1 | 구직자 | "서류 업로드" 버튼을 클릭한다 |
| 2 | 구직자 | PDF 파일을 선택하여 업로드한다 |
| 3 | 시스템 | PDF에서 텍스트를 추출한다 (pdfplumber) |
| 4 | 시스템 | 추출된 텍스트를 LLM에 전달하여 구조화 데이터로 변환한다 |
| 5 | 시스템 | 구조화 결과를 화면에 표시한다: 보유기술, 프로젝트경험, 성과지표, 소프트스킬, 키워드 |
| 6 | 구직자 | 파싱 결과를 확인하고, 필요 시 수정한다 |
| 7 | 시스템 | 최종 구조화 데이터를 저장한다 |

**대안 플로우**

| ID | 조건 | 행동 |
|----|------|------|
| A1 | 6단계에서 구직자가 파싱 결과를 수정함 | 수정된 데이터를 최종 데이터로 저장 |
| A2 | 3단계에서 PDF 텍스트 추출 실패 (이미지 기반 PDF) | "텍스트 기반 PDF만 지원됩니다" 안내 메시지 표시 |

**예외 플로우**

| ID | 조건 | 행동 |
|----|------|------|
| E1 | 파일 크기 초과 (10MB 이상) | "파일 크기는 10MB 이하만 가능합니다" 오류 표시 |
| E2 | PDF가 아닌 파일 형식 | "PDF 파일만 업로드 가능합니다" 오류 표시 |
| E3 | LLM 구조화 추출 실패 | "분석 중 오류가 발생했습니다. 다시 시도해주세요" 표시 + 재시도 버튼 |

**사후 조건**
- 서류의 구조화 데이터가 DB에 저장된다.
- 원문 텍스트는 사용자가 "저장" 선택 시에만 암호화 보관되며, 미선택 시 분석 완료 후 삭제된다.

### UC-02: 공고 입력 및 파싱

**액터**: 구직자, 기업 HR
**우선순위**: P0 (MVP 필수)

**사전 조건**
- 사용자가 분석할 공고의 텍스트를 확보하고 있다.

**기본 플로우**

| 단계 | 액터 | 행동 |
|------|------|------|
| 1 | 사용자 | "공고 입력" 영역에 공고 텍스트를 붙여넣는다 |
| 2 | 사용자 | "분석" 버튼을 클릭한다 |
| 3 | 시스템 | 공고 텍스트를 LLM에 전달하여 구조화 데이터로 변환한다 |
| 4 | 시스템 | 구조화 결과를 화면에 표시한다: 필수기술, 우대기술, 핵심업무, 요구경험, 조직문화키워드 |
| 5 | 사용자 | 파싱 결과를 확인하고, 필요 시 수정한다 |
| 6 | 시스템 | 최종 구조화 데이터를 저장한다 |

**대안 플로우**

| ID | 조건 | 행동 |
|----|------|------|
| A1 | 구직자가 복수의 공고를 입력 (최대 5개) | 각 공고를 개별 파싱 후 목록으로 저장 |

**예외 플로우**

| ID | 조건 | 행동 |
|----|------|------|
| E1 | 입력 텍스트가 100자 미만 | "공고 내용이 너무 짧습니다. 전체 공고를 입력해주세요" 안내 |
| E2 | 입력 텍스트가 채용 공고가 아닌 것으로 판단됨 | "채용 공고 형식이 아닌 것 같습니다. 확인 후 다시 입력해주세요" 안내 |

**사후 조건**
- 공고의 구조화 데이터가 DB에 저장된다.

### UC-03: Fit-Gap 분석 실행

**액터**: 구직자, 기업 HR
**우선순위**: P0 (MVP 필수)

**사전 조건**
- 서류 파싱 완료 (UC-01 사후 조건 충족)
- 공고 파싱 완료 (UC-02 사후 조건 충족)

**기본 플로우**

| 단계 | 액터 | 행동 |
|------|------|------|
| 1 | 사용자 | 분석할 서류와 공고 조합을 선택하고 "분석 시작"을 클릭한다 |
| 2 | 시스템 | 서류/공고 구조화 데이터의 각 항목을 임베딩으로 변환한다 |
| 3 | 시스템 | 공고 요구 항목 ↔ 서류 역량 항목 간 유사도를 비교한다 |
| 4 | 시스템 | 유사도 결과를 기반으로 Fit/Gap/Over를 분류한다 |
| 5 | 시스템 | 항목별 가중치를 적용하여 적합도 점수(0~100)를 산출한다 |
| 6 | 시스템 | LLM을 호출하여 원문 인용 기반 설명문을 생성한다 |
| 7 | 시스템 | 각 항목의 확신도(High/Medium/Low)를 결정한다 |
| 8 | 시스템 | 분석 결과를 DB에 저장하고 결과 화면으로 이동한다 |

**대안 플로우**

| ID | 조건 | 행동 |
|----|------|------|
| A1 | 기업 HR이 복수 서류를 일괄 분석 | 각 서류에 대해 Step 2~8을 반복 수행 후 리스트로 결과 표시 |

**예외 플로우**

| ID | 조건 | 행동 |
|----|------|------|
| E1 | LLM API 호출 실패 | "분석 중 오류가 발생했습니다" 표시 + 재시도 버튼 |
| E2 | 임베딩 API 호출 실패 | LLM 기반 텍스트 매칭으로 폴백 |
| E3 | 분석 시간 30초 초과 | "분석에 시간이 걸리고 있습니다. 잠시만 기다려주세요" 표시 (최대 60초) |

**사후 조건**
- 분석 결과(점수, Fit/Gap/Over, 설명, 확신도)가 DB에 저장된다.
- 신호등 분류가 결정된다 (🔴/🟡/🟢).

### UC-04: 분석 결과 조회

**액터**: 구직자, 기업 HR
**우선순위**: P0 (MVP 필수)

**사전 조건**
- Fit-Gap 분석이 완료됨 (UC-03 사후 조건 충족)

**기본 플로우 — 구직자 뷰**

| 단계 | 액터 | 행동 |
|------|------|------|
| 1 | 시스템 | 적합도 점수(0~100)와 신호등(🔴/🟡/🟢)을 상단에 표시한다 |
| 2 | 시스템 | 항목별 점수를 차트(바/레이더)로 표시한다: 필수기술, 우대사항, 직무경험, 소프트스킬, 성과근거 |
| 3 | 시스템 | 강점 TOP 3 카드를 표시한다 (원문 인용 포함) |
| 4 | 시스템 | 부족 TOP 3 카드를 표시한다 (원문 인용 + 개선 제안) |
| 5 | 시스템 | 과잉(Over) 항목을 표시한다 (해당 시) |
| 6 | 시스템 | 종합 설명문을 표시한다 |
| 7 | 시스템 | 확신도가 Low인 항목에는 "참고용" 표시를 추가한다 |
| 8 | 시스템 | "본 분석은 참고용이며, 최종 지원 결정은 사용자의 판단에 따릅니다" 면책 고지를 표시한다 |

**기본 플로우 — 기업 HR 뷰**

| 단계 | 액터 | 행동 |
|------|------|------|
| 1 | 시스템 | 지원자 리스트를 신호등 기준으로 정렬하여 표시한다 (🟢 → 🟡 → 🔴) |
| 2 | 시스템 | 각 지원자 카드에 적합도 점수 + 부족 사유 1줄 요약을 표시한다 |
| 3 | 기업 HR | 특정 지원자를 클릭하여 상세 분석 결과를 조회한다 |
| 4 | 시스템 | 구직자 뷰와 동일한 상세 결과를 표시한다 |

**사후 조건**
- 없음 (조회만 수행)

### UC-05: 서류 개선 제안 조회

**액터**: 구직자
**우선순위**: P1 (MVP 포함)

**사전 조건**
- Fit-Gap 분석 결과에 Gap 항목이 1개 이상 존재한다.

**기본 플로우**

| 단계 | 액터 | 행동 |
|------|------|------|
| 1 | 구직자 | 분석 결과 화면에서 "개선 제안" 탭/섹션을 선택한다 |
| 2 | 시스템 | 각 Gap 항목별로 개선 제안을 표시한다 |
| 3 | 시스템 | 제안의 형식: [Gap 항목명] + [공고 원문 인용] + [현재 서류 상태] + [구체적 개선 방향] |
| 4 | 구직자 | 개선 제안을 확인하고 서류 수정에 반영한다 |

**예시 출력**

```
[Gap] 협업 경험
  공고: "크로스펑셔널 팀과의 협업 경험 필수" (필수 요건 3번)
  현재: 4개 프로젝트 중 팀 협업 언급 1회, 구체적 방식 서술 0회
  제안: 프로젝트 경험에서 코드 리뷰, 스프린트 회의,
       타 직군(디자인/기획)과의 협업 사례를 추가로 서술하세요.
```

**사후 조건**
- 없음 (조회만 수행)

### UC-07: 공고 개선 인사이트 조회

**액터**: 기업 HR
**우선순위**: P0 (MVP 필수)

**사전 조건**
- 공고 파싱이 완료되어 있다 (UC-02 사후 조건 충족).

**기본 플로우**

| 단계 | 액터 | 행동 |
|------|------|------|
| 1 | 기업 HR | 공고 등록 후 "공고 분석" 버튼을 클릭한다 |
| 2 | 시스템 | 공고 텍스트를 분석하여 인사이트를 생성한다 |
| 3 | 시스템 | 인사이트 카드 2~3개를 표시한다 |
| 4 | 기업 HR | 인사이트를 확인하고, 필요 시 공고를 수정한다 |

**인사이트 유형**

| 유형 | 예시 |
|------|------|
| 필수 요건 과다 | "필수 요건 4개(Spring Boot, Redis, AWS, Spring Batch)는 경력 3년차 수준입니다. 신입 대상이라면 2개로 축소를 권장합니다." |
| 신입/경력 불일치 | "공고에 '신입 환영'이라 했으나, 요구 기술 수준은 경력 2~3년에 해당합니다." |
| 매력도 부족 | "복지/문화 관련 서술이 1줄입니다. 해당 내용을 보강하면 지원자 풀이 넓어질 수 있습니다." |

**사후 조건**
- 인사이트가 DB에 저장된다.

**사전 조건**

| 2 | 사용자 | 👍 또는 👎를 선택한다 |
| 3-a | 시스템 | (👍인 경우) "감사합니다!" 메시지를 표시하고 피드백을 저장한다 |
| 3-b | 시스템 | (👎인 경우) "어떤 부분이 부정확했나요?" 선택지를 표시한다 |

- "공고 요구사항을 잘못 파악함"
- "점수가 실제와 맞지 않음"
- "기타" (자유 입력)

**사후 조건**
- 피드백이 DB에 저장된다.

### UC-09: 합격 가능성 시뮬레이션

**액터**: 구직자
**우선순위**: P1 (확장)

**사전 조건**
- Fit-Gap 분석 결과가 존재한다.
- Gap 항목이 1개 이상 존재한다.

**기본 플로우**

| 단계 | 액터 | 행동 |
|------|------|------|
| 1 | 구직자 | 분석 결과 화면에서 "시뮬레이션" 버튼을 클릭한다 |
| 2 | 시스템 | 현재 Gap 항목 목록과 각 항목의 예상 점수 변화를 표시한다 |
| 3 | 구직자 | 보완할 Gap 항목을 선택한다 (예: "Redis 경험 추가") |
| 4 | 시스템 | 해당 항목이 Fit으로 전환되었다고 가정하고 점수를 재계산한다 |
| 5 | 시스템 | Before/After 점수를 비교 표시한다 (예: 68% → 85%) |
| 6 | 구직자 | 여러 조합을 시도하여 가장 효과적인 개선 방향을 파악한다 |

**사후 조건**
- 시뮬레이션 결과는 일시적이며 DB에 저장되지 않는다.

### UC-10: 피드백 레터 생성

**액터**: 기업 HR
**우선순위**: P2 (확장)

**사전 조건**
- 지원자에 대한 Fit-Gap 분석이 완료되어 있다.
- 해당 지원자가 🔴 부적합 또는 🟡 보류로 분류되어 있다.

**기본 플로우**

| 단계 | 액터 | 행동 |
|------|------|------|
| 1 | 기업 HR | 지원자 리스트에서 탈락 처리할 지원자를 선택한다 |
| 2 | 기업 HR | "피드백 메일 생성" 버튼을 클릭한다 |
| 3 | 시스템 | 해당 지원자의 Fit-Gap 분석 결과를 기반으로 정중한 거절 메일 초안을 생성한다 |
| 4 | 시스템 | 메일 초안을 미리보기로 표시한다 |
| 5 | 기업 HR | 초안을 확인하고 필요 시 수정한다 |
| 6 | 기업 HR | "발송" 또는 "복사" 버튼을 클릭한다 |

**피드백 레터 포함 내용**
- 지원에 대한 감사 인사
- 불합격 안내
- 분석 기반 구체적 피드백 1~2가지 (강점 + 개선점)
- 재지원 가능성 또는 격려 문구

**사후 조건**
- 생성된 피드백 레터가 DB에 기록된다.
- (발송 선택 시) 지원자에게 메일이 발송된다.
